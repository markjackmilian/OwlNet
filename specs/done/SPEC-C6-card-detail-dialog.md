# SPEC: Card Detail Dialog

> **Status:** Done
> **Created:** 2026-03-12
> **Author:** owl-planner + user
> **Priority:** High
> **Estimated Complexity:** L

## Context

The card detail dialog is the primary interface for viewing and interacting with a single card. It replaces the "Coming soon" placeholder introduced in SPEC-project-board-page. The dialog surfaces all card data in a structured layout: editable fields, a comment thread (human + agent), and a list of attachments generated by workflow triggers.

The dialog is opened by clicking a card on the board (SPEC-C4). It is a `MudDialog` — not a separate page — to keep the board context visible behind it.

## Actors

| Actor | Description |
|-------|-------------|
| **User** | Authenticated user viewing, editing, and commenting on a card |
| **System** | Loads card data, persists changes, displays agent-generated content |

## Functional Requirements

### Dialog Layout

1. The dialog SHALL be a `MudDialog` opened in a wide format (e.g., `MaxWidth.Large`, `FullWidth="true"`).
2. The dialog SHALL have a **header** containing:
   - Card number (e.g., `#42`) as a muted label.
   - Card title as the main heading (editable inline — see requirement 8).
   - A close button (top-right).
3. The dialog body SHALL be split into two panels:
   - **Left panel** (wider, ~65%): Title (inline edit), Description (inline edit), Comments section, Attachments section.
   - **Right panel** (narrower, ~35%): Status, Priority, Tags, Status History, metadata (CreatedAt, UpdatedAt).
4. The dialog SHALL NOT have explicit "Save" / "Cancel" buttons for field edits — changes are auto-saved on blur (per field). See requirement 9.

### Right Panel — Card Metadata

5. **Status**: displayed as a `MudSelect` dropdown populated from `GetProjectStatusesQuery`. Changing the status SHALL immediately call `ChangeCardStatusCommand` (with `ChangeSource = Manual`). On success, the status updates in the UI and the workflow trigger evaluation pipeline is invoked (SPEC-C8, fire-and-forget). On failure, the status reverts and a snackbar error is shown.
6. **Priority**: displayed as a `MudSelect` or segmented button. Changing the priority SHALL immediately call `UpdateCardCommand`. On failure, the value reverts and a snackbar error is shown.
7. **Tags**: displayed as a chip list. Each chip shows the tag name and color. An "Add tag" chip button opens a popover with the project's tag vocabulary for selection. Clicking an existing tag chip removes it from the card. All tag changes call `AddTagToCardCommand` or `RemoveTagFromCardCommand` immediately.
8. **Status History**: a collapsible section showing the card's status transitions ordered by most recent first. Each entry shows: previous status → new status, actor (user ID or agent name), change source (Manual/Trigger), and timestamp. Uses `GetCardStatusHistoryQuery`.

### Left Panel — Editable Fields

9. **Title**: displayed as a large text input. On blur, if the value has changed, the system SHALL call `UpdateCardCommand`. A subtle loading indicator appears during the save. On failure, the title reverts to its last saved value and a snackbar error is shown.
10. **Description**: displayed as a `MudTextField` with `Lines` set for multi-line input. The user edits raw Markdown. On blur, if the value has changed, the system SHALL call `UpdateCardCommand`. Below the textarea, a **Markdown preview** toggle button allows the user to switch between edit mode and rendered preview mode.
11. Auto-save on blur applies to both Title and Description. There is no explicit "Save" button for these fields.

### Comments Section

12. The comments section SHALL be displayed below the description in the left panel, under a "Comments" heading.
13. Comments SHALL be displayed in chronological order (oldest first), as a vertical thread.
14. Each comment SHALL display:
    - **Author indicator**: for human comments, a user avatar/icon and the author ID (or display name if available); for agent comments, a distinct robot/agent icon and the agent name.
    - **Timestamp**: relative time (e.g., "2 hours ago") with full timestamp on hover.
    - **Content**: rendered as Markdown.
    - **Source badge** (agent comments only): if the comment was generated by a workflow trigger, display a small badge showing the trigger name (or "Workflow" if trigger was deleted).
15. Human comments and agent comments SHALL be visually distinct (e.g., different background color or left border color).
16. At the bottom of the comments section, a **comment input area** SHALL allow the user to add a new human comment:
    - A `MudTextField` with `Lines="3"` for multi-line input (Markdown supported).
    - A "Post comment" button that calls `AddHumanCommentCommand`.
    - The button SHALL be disabled when the input is empty or while the request is in progress.
    - On success, the new comment appears at the bottom of the thread and the input is cleared.
    - On failure, a snackbar error is shown and the input is retained.
17. Comments are immutable — no edit or delete actions are shown.

### Attachments Section

18. The attachments section SHALL be displayed below the comments section in the left panel, under an "Attachments" heading.
19. If there are no attachments, the section SHALL display a muted hint: "No attachments yet. Attachments are generated automatically by workflow triggers."
20. Each attachment SHALL be displayed as a card/row showing:
    - File name (e.g., `code-review-summary.md`).
    - Agent name that generated it.
    - Trigger name (or "Workflow" if trigger was deleted).
    - Creation timestamp.
    - An **"View"** button that expands the attachment content inline (or opens a sub-dialog) rendering the full Markdown content.
21. Attachment content SHALL be loaded on demand (only when the user clicks "View") via `GetCardAttachmentContentQuery`. It SHALL NOT be loaded with the initial dialog load.
22. The expanded attachment view SHALL render the content as Markdown (full rendering, no line-clamp).
23. Attachments are immutable — no edit or delete actions are shown.

### Dialog Loading

24. When the dialog opens, the system SHALL load the following in parallel:
    - Card data (from the `CardDto` already available on the board — no extra query needed for basic fields).
    - Comments via `GetCardCommentsQuery`.
    - Attachments metadata (without content) via `GetCardAttachmentsQuery`.
    - Status history via `GetCardStatusHistoryQuery`.
    - Project statuses via `GetProjectStatusesQuery` (for the status dropdown).
    - Project tags via `GetProjectTagsQuery` (for the tag selector).
25. While loading, the dialog SHALL display skeleton loaders in the comments, attachments, and history sections.

### Dialog Close and Board Refresh

26. Closing the dialog SHALL notify the parent board component if any card data was changed (title, description, priority, status, tags), triggering a board refresh for the affected card.
27. The dialog SHALL be closable via the close button, the Escape key, or clicking outside (standard MudDialog behavior).

## User Flow

### View Card Details
1. User clicks on a card on the board.
2. Dialog opens; skeleton loaders appear in secondary sections.
3. Card title, description, status, priority, and tags are immediately visible (from `CardDto`).
4. Comments, attachments, and status history load asynchronously.

### Edit Title
1. User clicks on the title in the dialog.
2. Title becomes an editable text input.
3. User modifies the title and clicks elsewhere (blur).
4. System calls `UpdateCardCommand`; loading indicator appears briefly.
5. Title is saved; no snackbar on success (silent save).

### Change Status from Dialog
1. User opens the Status dropdown in the right panel.
2. User selects "Review".
3. System calls `ChangeCardStatusCommand`.
4. Status updates in the UI.
5. Trigger evaluation pipeline is invoked asynchronously (SPEC-C8).

### Add a Comment
1. User types a comment in the input area.
2. User clicks "Post comment".
3. System calls `AddHumanCommentCommand`.
4. New comment appears at the bottom of the thread.
5. Input is cleared.

### View an Attachment
1. User sees an attachment row: `code-review-summary.md` by `owl-reviewer`.
2. User clicks "View".
3. System calls `GetCardAttachmentContentQuery`.
4. Content renders as Markdown inline below the attachment row.

## Edge Cases and Error Handling

| Scenario | Expected Behavior |
|----------|-------------------|
| Auto-save fails (title/description) | Field reverts to last saved value; snackbar error shown |
| Status change fails | Status dropdown reverts; snackbar error shown |
| Priority change fails | Priority reverts; snackbar error shown |
| Tag add/remove fails | Tag state reverts; snackbar error shown |
| Comment submission fails | Input retained; snackbar error shown |
| Attachment content load fails | Error message shown inline where content would appear |
| Card is deleted while dialog is open | Dialog closes automatically; snackbar: "This card has been deleted." (future concern — not required in this spec) |
| Project has no tags | "Add tag" popover shows empty state: "No tags defined. Add tags in Project Settings." |
| Status history is empty | Section shows: "No status changes recorded." |
| No attachments | Section shows muted hint message |
| No comments | Section shows muted hint: "No comments yet. Be the first to comment." |

## Out of Scope

- Card deletion from the detail dialog (can be added in a future spec).
- Card duplication.
- Assigning a card to a user.
- Setting a due date.
- Linking cards to each other (dependencies).
- Exporting card content.
- Real-time updates (another user editing the same card simultaneously).
- Editing or deleting comments.
- Editing or deleting attachments.

## Acceptance Criteria

- [ ] Dialog opens on card click from the board.
- [ ] Dialog displays card number, title, description, status, priority, and tags.
- [ ] Title and description are editable inline with auto-save on blur.
- [ ] Auto-save failure reverts the field and shows a snackbar error.
- [ ] Status dropdown calls `ChangeCardStatusCommand` on change; reverts on failure.
- [ ] Status change triggers workflow trigger evaluation (SPEC-C8, fire-and-forget).
- [ ] Priority selector calls `UpdateCardCommand` on change; reverts on failure.
- [ ] Tags are displayed as chips; adding/removing calls the appropriate tag command immediately.
- [ ] Status history section displays transitions ordered by most recent first.
- [ ] Comments are displayed in chronological order with correct author indicators.
- [ ] Human and agent comments are visually distinct.
- [ ] Agent comments show trigger name badge (null-safe).
- [ ] Comment input allows posting a new human comment via `AddHumanCommentCommand`.
- [ ] Comment input is disabled while empty or request is in progress.
- [ ] Attachments list shows metadata (no content on initial load).
- [ ] "View" button loads and renders attachment content on demand.
- [ ] Skeleton loaders are shown during async data loading.
- [ ] Dialog close notifies the board to refresh if card data changed.
- [ ] Markdown preview toggle works for the description field.

## Open Questions

1. **Auto-save UX**: Should there be a subtle "Saved" confirmation indicator (e.g., a green checkmark that fades) after a successful auto-save, or should successful saves be completely silent? Currently specified as silent.
2. **Description edit/preview toggle**: Should the description start in preview mode (rendered Markdown) and switch to edit mode on click, or always start in edit mode? Starting in preview mode is more readable but requires an extra click to edit.
3. **Comment pagination**: If a card accumulates hundreds of comments, should they be paginated or virtualized? Currently all comments are loaded at once (per SPEC-C2). This can be revisited if performance becomes an issue.
