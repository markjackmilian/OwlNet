# SPEC: Card Attachments

> **Status:** Todo
> **Created:** 2026-03-12
> **Author:** owl-planner + user
> **Priority:** High
> **Estimated Complexity:** S

## Context

During workflow trigger execution, AI agents produce Markdown documents as output — analysis reports, code review summaries, implementation plans, etc. These documents need to be attached to the card that triggered the workflow, so the user can access them directly from the card's detail view.

Attachments are always generated by agents as part of a workflow trigger execution. Manual upload by users is explicitly out of scope. The full content of each attachment is stored in the database as text (no filesystem dependency).

This spec covers the domain entity, persistence, and Application layer (CQRS). The UI for displaying attachments is covered in SPEC-C6 (Card Detail Dialog). The actual agent execution that produces attachments is covered in SPEC-C8 (Trigger Execution).

## Actors

| Actor | Description |
|-------|-------------|
| **Agent** | AI agent that produces a Markdown document during workflow trigger execution |
| **System** | Persists the attachment and enforces validation rules |
| **User** | Read-only consumer of attachments via the card detail dialog |

## Functional Requirements

### CardAttachment Entity

1. The system SHALL persist card attachments as domain entities with the following properties:
   - **Id**: Guid, unique identifier, generated on creation.
   - **CardId**: foreign key to `Card`. Required.
   - **FileName**: string, required, 1–200 characters, not blank. A human-readable name for the attachment (e.g., `code-review-summary.md`). Does not need to be unique within a card.
   - **Content**: string, required, not blank. The full Markdown content of the attachment. No maximum length enforced at the domain level (stored as `TEXT`/`NVARCHAR(MAX)` in the database).
   - **AgentName**: string, required, 1–200 characters. The name of the agent that generated the attachment (filename without `.md` extension).
   - **WorkflowTriggerId**: Guid. Foreign key to `WorkflowTrigger`. Required at creation time.
   - **CreatedAt**: `DateTimeOffset`, set to UTC on creation.

2. `CardAttachment` records SHALL be **immutable** (append-only). Once created, no field SHALL be updatable.
3. A `CardAttachment` SHALL belong to exactly one `Card`.
4. Deleting a `Card` SHALL cascade-delete all its `CardAttachment` records.
5. Deleting a `WorkflowTrigger` SHALL NOT cascade-delete attachments that reference it — the `WorkflowTriggerId` FK SHALL be set to null on trigger deletion (set-null behavior).

### Card Entity — Attachment Navigation

6. The `Card` entity SHALL expose a navigation property `IReadOnlyList<CardAttachment> Attachments`.
7. The `Card` entity SHALL expose a method `AddAttachment(fileName, content, agentName, workflowTriggerId)` that:
   - Validates all required fields are non-null and non-blank.
   - Creates and appends a new `CardAttachment` to the collection.
   - Does NOT update `Card.UpdatedAt`.

### Application Layer — CQRS

8. The system SHALL implement the following command and queries for `CardAttachment`:
   - `AddCardAttachmentCommand` — CardId, FileName, Content, AgentName, WorkflowTriggerId → adds an attachment. Returns the new attachment's `Id` (Guid).
   - `GetCardAttachmentsQuery` — CardId → returns `IReadOnlyList<CardAttachmentDto>` ordered by `CreatedAt` ascending.
   - `GetCardAttachmentContentQuery` — AttachmentId → returns the full `Content` string of a single attachment (for on-demand loading).

9. The `CardAttachmentDto` record SHALL contain:
   - `Id` (Guid)
   - `CardId` (Guid)
   - `FileName` (string)
   - `AgentName` (string)
   - `WorkflowTriggerId` (Guid?, null if trigger was deleted)
   - `WorkflowTriggerName` (string?, denormalized name of the trigger; null if trigger was deleted)
   - `CreatedAt` (`DateTimeOffset`)
   - **Note**: `Content` is intentionally excluded from `CardAttachmentDto` to avoid loading large text in list views. Content is fetched separately via `GetCardAttachmentContentQuery`.

10. `AddCardAttachmentCommand` SHALL validate:
    - `CardId` references an existing card.
    - `FileName` is not blank and does not exceed 200 characters.
    - `Content` is not blank.
    - `AgentName` is not blank and does not exceed 200 characters.
    - `WorkflowTriggerId` references an existing workflow trigger.

### Persistence

11. The system SHALL add an EF Core configuration for `CardAttachment` using Fluent API exclusively.
12. Cascade delete from `Card` to `CardAttachment` SHALL be configured.
13. The `WorkflowTriggerId` FK SHALL use set-null on trigger deletion.
14. The `Content` column SHALL be mapped to an unbounded text type (`TEXT` for SQLite, `NVARCHAR(MAX)` for SQL Server).
15. The system SHALL generate dual-provider migrations (SQLite + SQL Server) for the new `CardAttachments` table.
16. The `CardAttachments` table SHALL have an index on `CardId` to support efficient retrieval by card.

## User Flow

*Note: The UI for this feature is covered in SPEC-C6. This spec covers only the domain and application layer.*

### Conceptual Flow — Agent Creates Attachment
1. A workflow trigger fires during a card status change (SPEC-C8).
2. The agent produces a Markdown document as output.
3. The trigger execution engine calls `AddCardAttachmentCommand` with the card ID, a generated filename, the Markdown content, the agent name, and the trigger ID.
4. The attachment is persisted in the database.
5. The user can view the attachment in the card's detail dialog (SPEC-C6).

## Edge Cases and Error Handling

| Scenario | Expected Behavior |
|----------|-------------------|
| `FileName` is blank | Reject with validation error: "File name cannot be empty." |
| `Content` is blank | Reject with validation error: "Attachment content cannot be empty." |
| `AgentName` is blank | Reject with validation error: "Agent name cannot be empty." |
| `WorkflowTriggerId` references a non-existent trigger | Reject with validation error: "Workflow trigger not found." |
| Card does not exist | Reject with not-found error. |
| Workflow trigger is deleted after attachment was created | Attachment is retained; `WorkflowTriggerId` is set to null; `WorkflowTriggerName` in DTO returns null. |
| Card is deleted | All attachments are cascade-deleted. |
| Very large Markdown content (e.g., 500KB) | Accepted and stored. No size limit enforced at the domain level. |

## Out of Scope

- Manual file upload by users.
- Non-Markdown attachment types (PDF, images, etc.).
- Editing or deleting attachments (immutable by design).
- Attachment versioning (multiple versions of the same document).
- Attachment download as a file (content is displayed inline in the UI).
- Full-text search within attachment content.
- UI for displaying attachments (covered in SPEC-C6).
- Agent execution that produces attachments (covered in SPEC-C8).

## Acceptance Criteria

- [ ] `CardAttachment` entity is persisted with all specified properties (Id, CardId, FileName, Content, AgentName, WorkflowTriggerId, CreatedAt).
- [ ] Attachments are immutable — no update or delete commands exist.
- [ ] `Card.AddAttachment()` validates required fields and appends the attachment.
- [ ] `WorkflowTriggerId` is nullable and uses set-null on trigger deletion.
- [ ] Card deletion cascades to all its attachments.
- [ ] `AddCardAttachmentCommand` is implemented and tested (happy path + validation failures).
- [ ] `GetCardAttachmentsQuery` returns attachments ordered by `CreatedAt` ascending, without `Content`.
- [ ] `GetCardAttachmentContentQuery` returns the full content of a single attachment.
- [ ] `CardAttachmentDto` excludes `Content` (list view is lightweight).
- [ ] `CardAttachmentDto` includes denormalized `WorkflowTriggerName` (null-safe).
- [ ] EF Core configuration uses Fluent API exclusively.
- [ ] `Content` column is mapped to unbounded text type for both providers.
- [ ] Index on `CardId` is defined.
- [ ] Dual-provider migrations (SQLite + SQL Server) are generated and applied cleanly.

## Dependencies

- **SPEC-WF2-board-card-entity** — `Card` entity must exist.
- **SPEC-WF3-workflow-trigger-entity** — `WorkflowTrigger` entity must exist (for FK reference).
- **EF Core dual-provider migrations** — new table `CardAttachments`.

## Open Questions

1. **FileName uniqueness**: Should `FileName` be unique within a card, or can the same filename appear multiple times (e.g., two runs of the same trigger producing `code-review.md`)? Currently not enforced — duplicates are allowed, distinguished by `CreatedAt`.
2. **Content size limit**: Should a maximum content size be enforced at the application layer (e.g., 1MB) to prevent abuse, even though no domain limit exists? This could be added as a validator guard.
