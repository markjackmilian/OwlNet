# SPEC-006: OpenCode Messaging

> **Status:** Todo
> **Created:** 2026-03-07
> **Author:** owl-planner + user
> **Priority:** High

## Context

The core interaction with OpenCode Server is sending prompts (messages) to the AI agent within a session and receiving responses. OpenCode Server supports synchronous prompt submission (`POST /session/:id/message`) that waits for the full AI response before returning, as well as asynchronous submission (`POST /session/:id/prompt_async`) that returns immediately.

For this phase, OwlNet uses the **synchronous approach** (complete response polling) — the system sends a prompt and waits for the full response. Real-time streaming of partial responses is out of scope (see SPEC-007 for future enhancement).

Messages in OpenCode are composed of **parts** (text, tool calls, tool results, etc.). The response includes both message metadata (`info`) and an array of `parts`.

## Actors

- **User** — sends prompts to the AI agent through the OwlNet Blazor UI.
- **Backend services** — DispatchR handlers that send prompts programmatically as part of automated workflows.
- **OpenCode Server** — processes prompts via the configured AI model and returns responses.

## Functional Requirements

1. The system SHALL extend `IOpenCodeClient` (or define a dedicated `IOpenCodeMessageService` interface in the Application layer) with methods for messaging.
2. The system SHALL support **sending a prompt** to a session by calling `POST /session/:id/message` with a body containing at minimum a `parts` array with a text part, and returning the full AI response.
3. The system SHALL support **sending a prompt asynchronously** by calling `POST /session/:id/prompt_async`, which returns immediately (HTTP 204) and the response can be retrieved later via events or message listing.
4. The system SHALL support **listing messages** in a session by calling `GET /session/:id/message`, returning an ordered collection of messages with their parts.
5. The system SHALL support **getting a specific message** by calling `GET /session/:id/message/:messageID`, returning the message info and parts.
6. The system SHALL define the following DTOs (records) in the Application layer:
   - `MessageDto` — represents a message with metadata (ID, role, creation timestamp, model info).
   - `MessagePartDto` — represents a part of a message (type: text, tool_call, tool_result, etc., plus content).
   - `MessageWithPartsDto` — composite record containing `MessageDto` and a collection of `MessagePartDto`.
   - `SendPromptRequest` — record for constructing a prompt (text content, optional model override, optional agent selection).
7. The system SHALL support specifying an optional **model override** when sending a prompt (provider ID + model ID), falling back to the server's default model if not specified.
8. The system SHALL support specifying an optional **agent** when sending a prompt (e.g., "build" or "plan").
9. The system SHALL enforce an extended HTTP timeout for prompt calls (default: 300 seconds / 5 minutes) since AI responses can take significant time, separate from the standard 30-second timeout for other API calls.
10. All messaging operations SHALL propagate `CancellationToken` and return `Result<T>` values.
11. All messaging operations SHALL log structured information: session ID, message ID, prompt length, response time, success/failure.
12. The system SHALL support **executing slash commands** within a session by calling `POST /session/:id/command` with a command name and arguments.

## User Flow

### Happy Path — Send a prompt and receive a response
1. The user has an active session (created via SPEC-005).
2. The user types a prompt in the OwlNet UI (e.g., "Explain the authentication flow in this project").
3. The system sends `POST /session/:id/message` with `{ parts: [{ type: "text", text: "..." }] }`.
4. The system shows a loading indicator while waiting for the response.
5. OpenCode Server processes the prompt and returns the full response.
6. The system displays the AI response (text parts) in the UI.

### Happy Path — List conversation history
1. The user opens an existing session.
2. The system calls `GET /session/:id/message` to load all messages.
3. The UI displays the conversation history: user prompts and AI responses in order.

### Happy Path — Send prompt with model override
1. The user selects a specific model from a dropdown before sending.
2. The system includes `model: { providerID: "anthropic", modelID: "claude-sonnet-4-20250514" }` in the request body.
3. The response is generated by the specified model.

## Edge Cases and Error Handling

| Scenario | Expected Behavior |
|----------|-------------------|
| OpenCode Server is not running | Return failure Result. UI shows connection error. |
| Session does not exist | Return failure Result. UI shows "Session not found". |
| Prompt is empty | Validate at service level. Return failure Result "Prompt text is required". |
| AI response times out (>5 min) | Return failure Result "Response timed out". UI shows timeout error with option to retry. |
| Server returns HTTP 429 (rate limited) | Return failure Result with rate limit details. UI shows "Rate limited, try again later". |
| Server returns partial/malformed response | Log warning, return failure Result. |
| AI response contains tool calls | Include tool call parts in the response DTO. The UI can display them as structured information. |
| Very long prompt text | No client-side limit (server handles its own limits). Log the prompt length. |
| CancellationToken is triggered during prompt | HttpClient cancels the request. Return cancelled Result. |

## Out of Scope

- Real-time streaming of partial AI responses (token-by-token). See SPEC-007.
- Structured output (JSON schema format) — can be added as an enhancement later.
- Image/file attachments in prompts.
- Permission request handling (`/session/:id/permissions/:permissionID`).
- Session revert/unrevert operations.
- Shell command execution via `/session/:id/shell`.
- UI components for the chat interface (will be a separate UI spec).

## Acceptance Criteria

- [ ] Messaging methods are defined in an interface in the Application layer.
- [ ] Implementation calls the correct OpenCode Server endpoints in the Infrastructure layer.
- [ ] `MessageDto`, `MessagePartDto`, `MessageWithPartsDto`, and `SendPromptRequest` records are defined.
- [ ] Send prompt calls `POST /session/:id/message` with correct body structure.
- [ ] Send prompt async calls `POST /session/:id/prompt_async`.
- [ ] List messages calls `GET /session/:id/message` and returns ordered collection.
- [ ] Get message by ID calls `GET /session/:id/message/:messageID`.
- [ ] Execute command calls `POST /session/:id/command`.
- [ ] Optional model override is supported in prompt requests.
- [ ] Optional agent selection is supported in prompt requests.
- [ ] Extended timeout (300s) is enforced for prompt calls.
- [ ] All methods propagate `CancellationToken` and return `Result<T>`.
- [ ] Structured logging with session ID, message ID, timing.
- [ ] Unit tests cover: send prompt, list messages, get message, command execution, timeout, error scenarios.
